 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ

 ### 1. SSRF-защита неполная — DNS Rebinding уязвимость

 src/dns.rs и src/ssrf.rs: проверка IP происходит только на момент DNS-резолва, но не при каждой попытке подключения. Злоумышленник
 может использовать DNS rebinding: сначала домен резолвится в публичный IP (проходит проверку), затем при реальном соединении — в
 приватный IP. reqwest не гарантирует, что использует только заранее резолвленные адреса. Нет повторной валидации IP в момент
 установления TCP-соединения.

 ### 2. SSRF-проверка не блокирует 0.0.0.0/8 целиком

 src/ssrf.rs — is_ip_safe использует addr.is_loopback(), что проверяет только 127.0.0.0/8. Но is_loopback() в std — это именно
 127.x.x.x. Это правильно, но is_link_local() проверяет только 169.254.0.0/16. Однако is_private() в std не включает 127.0.0.0/8. В
 целом покрытие хорошее, но:
 - IPv6 is_documentation() не проверяется — 2001:db8:: не блокируется
 - IPv6 is_multicast() не проверяется — потенциально опасные адреса

 ### 3. Заголовки от пользователя могут быть использованы для атак

 src/handlers.rs — параметр headers позволяет передавать произвольные заголовки. Хотя hop-by-hop заголовки фильтруются, можно
 передать Authorization, Cookie и другие чувствительные заголовки для любого URL. Нет валидации на предмет header injection
 (например, заголовки с \r\n).

 ### 4. Нет ограничения размера тела ответа

 src/handlers.rs — при скачивании изображения используется response.bytes().await, который загружает весь ответ в память. Даже при
 max_download_bytes = 0 (по умолчанию) можно вызвать DoS, заставив сервер скачать гигабайтный файл. Лимит проверяется после полной
 загрузки.

 ### 5. max_download_bytes = 0 по умолчанию = без лимита

 config.toml — max_download_bytes = 0. Это значит, что по умолчанию сервер может скачать файл любого размера. Для production-сервиса
 это опасно.

 🟡 СУЩЕСТВЕННЫЕ ПРОБЛЕМЫ

 ### 6. Память: двойное хранение изображений

 src/image.rs — изображение загружается через image::load_from_memory() (полный декод в RAM), затем ресайз создаёт ещё одну копию,
 затем энкодинг — ещё одну. Для 10MP+ изображений это может потребовать сотни мегабайт RAM на один запрос. Нет потоковой обработки.

 ### 7. Конвертация в RGB после B&W бессмысленна

 src/image.rs строки 186-195: после img.grayscale() (Luma8) вызывается img.into_rgb8(). Это создаёт RGB-копию из grayscale, которая
 потом кодируется в JPEG — в JPEG это будет 3 канала одинаковых значений. Работает, но неоптимально — можно кодировать grayscale
 напрямую в JPEG.

 ### 8. Потеря alpha-канала без предупреждения

 src/image.rs строка 195: img.into_rgb8() молча отбрасывает альфа-канал. PNG с прозрачностью превратится в JPEG с чёрным (или белым)
 фоном. Пользователь об этом не узнает. Нет лога о потере прозрачности.

 ### 9. WebP используется как формат по умолчанию, но webp crate — lossy only

 config.toml — default_jpeg = false значит по умолчанию WebP. Но webp crate (libwebp) кодирует только lossy. Нет поддержки lossless
 WebP. Нет параметра lossless в API.

 ### 10. OutputFormat::Png объявлен, но не используется

 src/image.rs строка 49: #[allow(dead_code)] Png — мёртвый код. Нет пути к PNG-энкодированию.

 ### 11. Нет rate limiting

 Нет ограничений на количество запросов от одного IP. Любой клиент может затопить сервер запросами.

 ### 12. Нет CORS-заголовков

 Сервис явно предназначен для работы с клиентским приложением (Komikku), но нет CORS headers. Браузерные клиенты не смогут обращаться
 к сервису напрямую.

 ### 13. health endpoint слишком простой

 src/handlers.rs — /health не проверяет состояние зависимостей. Нет readiness/liveness разделения.

 ### 14. Статистика сбрасывается при перезапуске

 src/handlers.rs — Stats хранится в памяти. При перезапуске сервера вся статистика теряется. Нет персистентности.

 ### 15. Нет graceful shutdown

 src/main.rs — сервер не обрабатывает SIGTERM/SIGINT. При перезапуске текущие запросы прерываются.

 ### 16. tokio = { features = ["full"] } — избыточные фичи

 Cargo.toml — включены все фичи tokio, включая tracing, signal, parking_lot и другие, которые могут не использоваться. Это
 увеличивает бинарь и время компиляции.

 🟠 КОСВЕТНЫЕ ПРОБЛЕМЫ / УЛУЧШЕНИЯ

 ### 17. quality валидация в handler'е дублирует проверку в from_query

 src/handlers.rs строка 101: filter(|q| *q >= 1 && *q <= 100) и строка 156: if quality < 1 || quality > 100 — дублирование. Если
 quality берётся из конфига, проверка на строке 156 сработает, но если из query — она уже отфильтрована. Логика работает, но
 запутанная.

 ### 18. Нет валидации resize параметра

 src/handlers.rs строка 109: resize парсится как u32, но может быть 0 (отключить) или 4294967295. Нет проверки на разумные значения.

 ### 19. Content-Type оригинала игнорируется при возврате оригинала

 src/handlers.rs строка 235: при возврате оригинала используется content_type от upstream сервера, но при возврате сжатой версии —
 resolve_content_type игнорирует оригинальный content type. Это правильно в целом, но при возврате оригинала после ошибки
 декодирования (строка 209) используется оригинальный тип, что может быть неверным.

 ### 20. Нет поддержки Accept заголовка

 Клиент не может запросить предпочтительный формат через Accept. Формат определяется только через query params.

 ### 21. Нет HTTP caching заголовков

 Нет Cache-Control, ETag, Last-Modified. Для image proxy это критично — каждый запрос скачивает и перерабатывает изображение заново.

 ### 22. prefer_original_if_smaller сравнивает >= а не >

 src/handlers.rs строка 227: compressed_size >= original_size — если сжатое равно оригиналу по размеру, возвращается оригинал. Это
 логично, но значит, что при конвертации PNG→JPEG одинакового размера клиент получит PNG, а не JPEG.

 ### 23. Нет логирования ошибок с request ID

 Нет correlation ID / request ID для трейсинга запросов через лог. В production сложно отследить конкретный запрос.

 ### 24. Интеграционные тесты не покрывают SSRF

 tests/integration_test.rs — нет тестов на SSRF-защиту, на заголовки, на Content-Type, на лимиты размера.

 ### 25. Нет Dockerfile / docker-compose

 Нет контейнеризации. Для production-развёртывания это минус.

 ### 26. config crate конфликтует с модулем config

 src/config.rs — модуль называется config, но используется и crate config. В load_config — config::Config::builder() и config::File —
 это crate, а config::load_config — это модуль. Работает благодаря crate::config, но путает.

 ### 27. Нет README.md

 В репозитории нет документации для разработчиков.

 ### 28. Пароль прокси в логах?

 src/main.rs строка 44: proxy = ?cfg.proxy.as_ref().map(|p| &p.address) — логится только адрес, не пароль. Это хорошо. Но proxy_url
 строится с логином/паролем в открытом виде в строке — если эта строка попадёт в лог (например, при ошибке), пароль будет виден.

 ### 29. fast_image_resize конвертация обратно через from_raw

 src/image.rs строки 168-181: после ресайза данные конвертируются обратно через into_vec() → from_raw(). Это может потерпеть неудачу,
 если fast_image_resize вернёт данные в другом порядке пикселей. Тесты это не проверяют.

 ### 30. Нет поддержки ICC color profiles / EXIF

 При конвертации теряются метаданные изображения: EXIF, ICC profiles, ориентация. Изображение может быть перевернуто, если EXIF
 Orientation ≠ 1.
