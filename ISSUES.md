 🔴 Критические проблемы

 ### 1. SSRF (Server-Side Request Forgery)

 Файл: handlers.rs:257

 ```rust
   let mut request = client.get(url);
 ```

 URL из параметра ?url= используется без валидации. Злоумышленник может подставить:
 - file:///etc/passwd — чтение локальных файлов
 - http://169.254.169.254/... — доступ к metadata-сервису облака
 - http://internal-service:8080/... — сканирование внутренней сети

 Нет ни проверки схемы (только http/https), ни блокировки IP-диапазонов, ни DNS rebinding protection.

 ### 2. Отсутствие лимитов (DoS)

 - max_download_bytes = 0 по умолчанию — сервер скачивает файлы любого размера
 - Нет rate limiting — один клиент может забить сервер запросами
 - Нет ограничения на размер декодированного изображения — злоумышленник может отправить «image bomb» (например, PNG 10000×10000),
 что приведёт к OOM
 - Нет ограничения на количество одновременных запросов

 ### 3. Обманчивый параметр jpg / default_jpeg

 Файл: image.rs:103-112

 ```rust
   let output_format = if to_jpeg {
       JpegEncoder::new_with_quality(&mut cursor, quality.max(1))...
   } else {
       // Fallback на JPEG (WebP lossless в image 0.25 даёт большие файлы для фото)
       JpegEncoder::new_with_quality(&mut cursor, quality.max(1))...
   };
 ```

 Всегда кодируется в JPEG, независимо от значения to_jpeg. Варианты OutputFormat::WebP и OutputFormat::Png помечены
 #[allow(dead_code)] и никогда не используются. Клиент, передающий jpg=0, получает JPEG — бессмысленный параметр.

 🟡 Серьёзные недоработки

 ### 4. Отсутствие аутентификации

 Кто угодно, кто знает адрес сервера, может использовать его как прокси. Нет API ключей, токенов, или хотя бы IP-whitelist.

 ### 5. Нет graceful shutdown

 Файл: main.rs — нет обработки SIGTERM/SIGINT. Сервер не дожидается завершения активных запросов при остановке.

 ### 6. /health не проверяет реальное здоровье

 Файл: handlers.rs:218-223

 ```rust
   pub async fn handle_health() -> impl IntoResponse {
       Json(serde_json::json!({ "status": "ok", "service": "bh-rust" }))
   }
 ```

 Всегда возвращает ok, даже если прокси недоступен, память переполнена, или диск полон. Не подходит для orchestrator'ов (k8s, docker
 healthcheck).

 ### 7. Утечка внутренней информации в ответах об ошибках

 ```rust
   return (StatusCode::INTERNAL_SERVER_ERROR,
       format!("Failed to process image: {}", e)).into_response();
 ```

 Детали ошибок (путь к файлу, версия библиотеки и т.д.) уходят клиенту.

 ### 8. Нет CORS

 Если сервис используется из браузера (например, через пользовательский скрипт), CORS заголовки отсутствуют — запросы будут
 заблокированы.

 ### 9. Не обрабатываются анимированные изображения

 image::load_from_memory загружает только первый кадр анимированных GIF и мультикадровых TIFF. Анимация теряется без предупреждения.

 ### 10. prefer_original_if_smaller использует >= вместо >

 Файл: handlers.rs:173

 ```rust
   let return_original = cfg.prefer_original_if_smaller && compressed_size >= original_size;
 ```

 Если сжатый файл равен по размеру оригиналу — возвращается оригинал. Это означает, что изображение было декодировано и
 перекодировано зря (потеря качества при JPEG→JPEG), хотя отдаётся оригинал.

 🟠 Средние проблемы

 ### 11. Content-Length дублирует то, что axum делает сам

 Файл: handlers.rs:208-213

 ```rust
   ("Content-Length", response_data.len().to_string().as_str()),
 ```

 Axum автоматически добавляет Content-Length для Vec<u8>. Ручная установка избыточна.

 ### 12. Нет валидации URL как URL

 ```rust
   let url = map.get("url")?.clone();
 ```

 Мусорная строка типа ??? будет передана в client.get(), которая вернёт ошибку «invalid URL». Лучше валидировать заранее и вернуть
 понятное сообщение.

 ### 13. Нет request ID в логах

 При одновременных запросах невозможно отследить, какой лог-запись какому запросу принадлежит.

 ### 14. config::Serialize не нужен

 Файл: config.rs — AppConfig и все подструктуры деривят Serialize, но нигде не сериализуются. Лишний feature и код.

 ### 15. tokio features = ["full"] — слишком много

 Для HTTP-сервера не нужны все фичи tokio. Это увеличивает размер бинарника и время компиляции.

 ### 16. Статистика сбрасывается при рестарте

 Счётчики в AtomicU64 живут только в памяти. При рестарте сервера — всё обнуляется. Для мониторинга это плохо.

 ### 17. Пароль прокси в открытом виде

 Файл: config.rs:71

 ```rust
   pub password: Option<String>,
 ```

 Пароль хранится в TOML без шифрования. В логах при debug уровне будет выведен.

 ### 18. started_on — u64 секунд с эпохи

 Файл: main.rs:52

 ```rust
   .as_secs()
 ```

 Переполнится в 2106 году. Сейчас не критично, но SystemTime можно хранить как-is.

 ### 19. quality.max(1) — тихое исправление

 Файл: image.rs:104

 ```rust
   JpegEncoder::new_with_quality(&mut cursor, quality.max(1))
 ```

 Если quality = 0, тихо заменяется на 1. Лучше валидировать на входе и вернуть ошибку.

 ### 20. Нет проверки Content-Type исходного файла

 Сервер скачивает файл по URL и пытается декодировать как изображение, но не проверяет Content-Type ответа. Если URL ведёт на
 HTML-страницу, сервер всё равно попробует декодировать.

 ### 21. Content-Type с параметрами теряется

 Файл: handlers.rs:275-280

 ```rust
   .unwrap_or_else(|| "application/octet-stream".to_string());
 ```

 Если оригинальный Content-Type — image/jpeg; charset=..., он берётся целиком. Но при возврате оригинала этот Content-Type с
 параметрами может быть некорректно обработан клиентом.

 🟢 Мелкие замечания

 ### 22. unwrap() в тестах

 Файл: integration_test.rs — unwrap() без контекста. При падении теста непонятно, что именно не сработало.

 ### 23. Нет тестов на обработку ошибок

 Тесты покрывают только happy path. Нет тестов на: невалидный URL, повреждённое изображение, превышение лимита, и т.д.

 ### 24. Нет .dockerignore / Dockerfile

 Для продакшена нужен контейнер. README упоминает openwrt/mipsel, но Dockerfile отсутствует.

 ### 25. bh-rust vs bh_rust — несогласованность

 Пакет называется bh-rust (дефис), но в коде — bh_rust (андерскор). Это нормально для Rust, но может сбивать с толку.
